# ⚡ Производительность

rt оптимизирован для эффективной обработки больших деревьев каталогов.

> **Среда тестирования:** Windows, NTFS, замер end-to-end (включая запуск процесса).  
> Время указано как медиана по Criterion / hyperfine.

---

## Параллельный режим

```bash
# Авто-определение потоков (по числу ядер ЦП)
rt --parallel

# Явное указание количества потоков
rt --parallel --threads 4

# Настройка ёмкости внутренней очереди
rt --parallel --queue-cap 8192
```

Параллельный режим использует Rayon с алгоритмом work-stealing для балансировки нагрузки.

> **Примечание:** параллельный режим эффективен начиная с ~1 000 файлов. На малых деревьях накладные расходы на потоки могут незначительно увеличить время выполнения.

### ⚙️ Влияние опций (10 000 файлов, seq)

| Опция                | Время    |
|----------------------|----------|
| plain                | 5.6 с    |
| `--sort size`        | 5.6 с    |
| `--sort mtime`       | 5.6 с    |
| `-C` (цвет)          | 5.6 с    |
| `--icons`            | 5.6 с    |
| `-d` (только дир.)   | 1.25 с   |
| `-L 2` (глубина 2)   | 75.5 мс  |
| `-L 3` (глубина 3)   | 255 мс   |

> Сортировка, цвет и иконки не замедляют работу. Ограничение глубины (`-L`) даёт кратное ускорение.

### 🌊 Streaming vs Standard (10 000 файлов)

| Режим                          | Время   |
|--------------------------------|---------|
| seq                            | 5.6 с   |
| streaming                      | 5.6 с   |
| par                            | 866 мс  |
| streaming + `--max-entries 100`| 53 мс   |
| seq + `--max-entries 100`      | 5.6 с   |

> **Ключевое отличие:** `--max-entries` в streaming-режиме обеспечивает раннюю остановку — обход прекращается после вывода N записей. В стандартном режиме дерево строится целиком, затем обрезается.

### 💾 Потребление памяти

Измерение: `PeakWorkingSet64` (Windows Task Manager → «Peak Working Set»).

| Файлов  | Sequential | Streaming | Parallel | Экономия streaming |
|---------|------------|-----------|----------|--------------------|
| 10 000  | 15.6 МБ    | 6.6 МБ    | 20.8 МБ  | **58%**                |
| 100 000 | 100.2 МБ   | 10.4 МБ   | 98.8 МБ  | **90%**                |
| 1 000 000 |  938.9 МБ  |  59.0 МБ  | 828 МБ   | **94%**              |

> Streaming-режим не строит дерево в памяти, а выводит записи по мере обхода. На больших деревьях экономия памяти составляет 90%+.

## Запуск бенчмарков

### Criterion (до 100k файлов, ~10–15 мин)

```bash
# Основной набор (100 / 10k / 100k + форматы + опции + streaming)
cargo bench --bench retree_perf

# Сохранение baseline для отслеживания регрессий
cargo bench --bench retree_perf -- --save-baseline main

# Сравнение с baseline
cargo bench --bench retree_perf -- --baseline main
```

### Criterion (1M файлов, ~2 часа)

```bash
cargo bench --bench retree_perf_xlarge
```

### Hyperfine (1M файлов, ~3–5 мин, рекомендуется)

```powershell
# Требует: winget install sharkdp.hyperfine
.\benches\bench_xlarge.ps1
```

### Замер памяти

```powershell
# Все размеры
.\benches\bench_memory.ps1

# Конкретные размеры
.\benches\bench_memory.ps1 -Sizes 10k,100k
.\benches\bench_memory.ps1 -Sizes 1m -Runs 5
```

## Управление тестовыми деревьями

Деревья создаются один раз в `target/bench_trees/` и переиспользуются между запусками.

```powershell
# Пересоздать все деревья
Remove-Item -Recurse target\bench_trees

# Пересоздать конкретное
Remove-Item -Recurse target\bench_trees\medium_10k
```

## HTML-отчёты Criterion

После запуска бенчмарков отчёт доступен по адресу:

```
target/criterion/report/index.html
```

Отчёт содержит: медиану, стандартное отклонение, графики распределения и регрессии.
